# テンプレート: Zenn記事

既存4記事で実証済みの構造。生成時は [../prompts/zenn-article.md](../prompts/zenn-article.md) と併用する。

## frontmatter

```yaml
---
title: 実数 + 何をしたか + 得られる示唆(例「Claude Code hooksを47本実装した話：AIへの自動指示を設計するという仕事」)
emoji: 🪝
type: tech            # tech / idea
topics: [claudecode, automation]   # 最大5個。人気タグと固有タグを混ぜる
published: false      # スケジューラが publish_scheduled で自動切替
publish_scheduled: '2026-XX-XXT09:00:00+09:00'
---
```

ドラフト運用時は `hold_reason` / `next_action` / `stage` を frontmatter に足してよい(公開前に削除)。

## 本文構造

```markdown
:::message
**📚 シリーズ名(該当する場合)**
相互リンク
:::

## TL;DR

- 結論と実数を3〜4点(この記事で一番おいしい情報をここで出し切る)

:::message
**検証環境**: 構成・時点(2026年X月時点)。仕様変動への注意と公式ドキュメントへの誘導
:::

:::details この記事の対象読者・前提・得られること
- **対象**: 誰向けか
- **前提**: 必要な予備知識
- **得られること**: 読後に何ができるようになるか
:::

---

## なぜ〇〇したのか(動機・失敗の自己開示から入る)

## 全体像(図解: Mermaid推奨。Zennで直描画できる)

## 本体(実装・手順。読者の疑問の順: 何を → どうやって → 落とし穴)

## うまくいかなかったこと・罠(必須節。信頼の源泉)

## 結果(Before/After を実数で)

## まとめ(次の記事・関連記事への導線)
```

## 執筆チェック(構造レベル)

- [ ] TL;DRだけ読んでも価値があるか
- [ ] 図解が最低1つあるか(「わかりやすい」評価の必須条件)
- [ ] 「うまくいかなかったこと」の節があるか
- [ ] 変わりやすい情報(価格・仕様)に時点表記があるか
- [ ] 著者の経験・判断が背骨になっているか(Zenn「人が主体」原則)
